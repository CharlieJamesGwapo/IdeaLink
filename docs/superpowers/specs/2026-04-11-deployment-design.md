# IdeaLink Deployment Design
**Date:** 2026-04-11  
**Approach:** Config-file driven (Option B)

## Architecture Overview

```
GitHub repo (main branch)
├── backend/          → Render Web Service (Go + Gin)
│   └── cmd/main.go   → build: go build -o server ./cmd/main.go
└── frontend/         → Netlify Site (React + Vite)
    └── dist/         → publish directory after: npm run build

Render PostgreSQL (managed DB, free tier)
  └── DATABASE_URL → injected into Render Web Service as env var
```

On every push to `main`, Render auto-rebuilds the Go backend and Netlify auto-rebuilds the frontend.

## Files to Create

### 1. `render.yaml` (repo root)
Defines the Render Web Service and PostgreSQL database as infrastructure-as-code.

- Runtime: Go
- Root directory: `backend/`
- Build command: `go build -o server ./cmd/main.go`
- Start command: `./server`
- Database: `idealink-db` (free tier managed PostgreSQL)
- `DATABASE_URL` injected automatically from the managed DB
- `JWT_SECRET` auto-generated by Render
- `FRONTEND_URL` set manually after Netlify URL is known

### 2. `netlify.toml` (repo root)
Defines Netlify build settings and SPA redirect rule.

- Base: `frontend/`
- Build command: `npm run build`
- Publish directory: `frontend/dist`
- Catch-all redirect: `/* → /index.html` (status 200) for React Router

### 3. `backend/internal/db/migrate.go`
Auto-runs SQL migrations on startup using the existing `001_initial.sql` file. Uses `CREATE TABLE IF NOT EXISTS` semantics so it is safe to run on every deploy. Called before the HTTP server starts in `main.go`.

### 4. `backend/cmd/main.go` (modified)
Calls the migration runner after DB connection is established, before registering routes and starting the server.

## Environment Variables

### Netlify Dashboard (set manually)
| Key | Value |
|-----|-------|
| `VITE_API_URL` | `https://idealink-backend.onrender.com` (Render service URL) |

### Render Dashboard (set manually after Netlify deploy)
| Key | Value |
|-----|-------|
| `FRONTEND_URL` | `https://your-site.netlify.app` (Netlify site URL) |

`DATABASE_URL` and `JWT_SECRET` are handled automatically by `render.yaml`.

## CORS
Already implemented — `middleware.CORS(cfg.FrontendURL)` reads `FRONTEND_URL` from env. No code changes needed beyond setting the correct value in Render.

## Deployment Flow
1. Add `render.yaml` + `netlify.toml` to repo and push to GitHub
2. In Render dashboard: **New → Blueprint** → connect repo → Render reads `render.yaml`, creates DB + web service automatically
3. In Netlify dashboard: **Add new site → Import from Git** → connect repo → Netlify reads `netlify.toml`
4. Set `VITE_API_URL` in Netlify environment variables → trigger redeploy
5. Set `FRONTEND_URL` in Render environment variables → Render redeploys
6. On first deploy, migration runner creates all tables automatically

## Database Migrations
Migrations run automatically on every backend startup via `backend/internal/db/migrate.go`. The migration reads `backend/internal/migrations/001_initial.sql` and executes it. Safe to re-run because all DDL uses `IF NOT EXISTS` guards.

## Constraints
- Render free tier web services spin down after 15 minutes of inactivity (cold start ~30s on first request)
- Render free PostgreSQL has a 90-day expiry — upgrade to paid if needed for production
- `JWT_SECRET` is auto-generated by Render on first deploy; rotating it will invalidate all existing sessions
