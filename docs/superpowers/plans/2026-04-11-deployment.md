# IdeaLink Deployment Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deploy the IdeaLink frontend to Netlify and backend (Go + PostgreSQL) to Render using config-file-driven infrastructure.

**Architecture:** `netlify.toml` at repo root tells Netlify how to build and serve the React/Vite frontend with SPA redirect rules. `render.yaml` at repo root defines the Go web service and managed PostgreSQL database as Render Blueprint infrastructure. Migrations already auto-run on startup via `config.ConnectDB` → `runMigrations` in `backend/internal/config/db.go` — no new code needed.

**Tech Stack:** React 19 + Vite (Netlify), Go 1.21 + Gin (Render Web Service), PostgreSQL (Render managed DB), render.yaml Blueprint, netlify.toml

## Task 1: Create `netlify.toml`

**Files:**
- Create: `netlify.toml` (repo root)

- [ ] **Step 1: Create the file**

```toml
[build]
  base    = "frontend"
  command = "npm run build"
  publish = "frontend/dist"

[[redirects]]
  from   = "/*"
  to     = "/index.html"
  status = 200
```

The `[[redirects]]` rule is critical — without it, any direct URL like `/admin` returns a 404 because Netlify tries to find a file at that path. With it, React Router handles all routing client-side.

## Task 2: Create `render.yaml`

**Files:**
- Create: `render.yaml` (repo root)

- [ ] **Step 1: Create the file**

```yaml
services:
  - type: web
    name: idealink-backend
    runtime: go
    rootDir: backend
    buildCommand: go build -o server ./cmd/main.go
    startCommand: ./server
    envVars:
      - key: DATABASE_URL
        fromDatabase:
          name: idealink-db
          property: connectionString
      - key: JWT_SECRET
        generateValue: true
      - key: FRONTEND_URL
        sync: false

databases:
  - name: idealink-db
    plan: free
```

Key notes:
- `DATABASE_URL` is automatically injected from the managed PostgreSQL `idealink-db` — no manual copy-paste needed.
- `JWT_SECRET` is auto-generated by Render on first deploy — rotating it invalidates all user sessions.
- `FRONTEND_URL` is `sync: false` — you set this manually in the Render dashboard after you know the Netlify URL. It controls the CORS `Access-Control-Allow-Origin` header in `backend/internal/middleware/cors.go:11`.
- `rootDir: backend` means Render runs all commands from the `backend/` directory.

## Task 4: Deploy Backend on Render

- [ ] **Step 1: Create a Blueprint in Render**

1. Go to [render.com/dashboard](https://render.com/dashboard)
2. Click **New** → **Blueprint**
3. Connect your GitHub repository (the IdeaLink repo)
4. Render will detect `render.yaml` automatically
5. Click **Apply** — Render will create:
   - `idealink-db` (PostgreSQL managed database, free tier)
   - `idealink-backend` (Go web service)

- [ ] **Step 2: Wait for first deploy to complete**

In the Render dashboard, open the `idealink-backend` service → **Logs** tab. Wait until you see:

```
Migrations applied
Server starting on :10000
```

(Render sets `PORT` to `10000` internally — this is fine, the backend reads it from env.)

- [ ] **Step 3: Note the backend URL**

In the `idealink-backend` service overview, copy the public URL. It looks like:
```
https://idealink-backend.onrender.com
```
(or a similar slug if the name was taken)

Keep this URL — you'll need it for the next two tasks.

## Task 5: Deploy Frontend on Netlify

- [ ] **Step 1: Create a new Netlify site**

1. Go to [app.netlify.com](https://app.netlify.com)
2. Click **Add new site** → **Import an existing project**
3. Select **GitHub** and connect the IdeaLink repository
4. Netlify detects `netlify.toml` — the build settings will auto-populate:
   - Base directory: `frontend`
   - Build command: `npm run build`
   - Publish directory: `frontend/dist`
5. Click **Deploy site**

- [ ] **Step 2: Set the `VITE_API_URL` environment variable**

1. In Netlify: **Site configuration** → **Environment variables** → **Add a variable**
2. Key: `VITE_API_URL`
3. Value: the Render backend URL from Task 4 Step 3, e.g. `https://idealink-backend.onrender.com`
4. Click **Save**

- [ ] **Step 3: Trigger a redeploy**

1. Go to **Deploys** tab
2. Click **Trigger deploy** → **Deploy site**
3. Wait for build to complete (usually 1-2 minutes)

- [ ] **Step 4: Note the Netlify site URL**

On the Netlify site overview, copy the URL. It looks like:
```
https://idealink.netlify.app
```
Keep this URL — you need it for the next task.

## Task 6: Wire CORS — Set `FRONTEND_URL` on Render

The backend CORS middleware (`backend/internal/middleware/cors.go:11`) sets `Access-Control-Allow-Origin` to exactly the value of `FRONTEND_URL`. It must match the Netlify URL exactly (no trailing slash).

- [ ] **Step 1: Set `FRONTEND_URL` in Render**

1. In Render dashboard → `idealink-backend` service → **Environment** tab
2. Find `FRONTEND_URL` (it was created as empty/unset from `render.yaml`)
3. Set value to your Netlify URL, e.g. `https://idealink.netlify.app`
4. Click **Save Changes** — Render will automatically redeploy the service

- [ ] **Step 2: Wait for Render redeploy**

Watch the Logs tab until you see `Server starting on :10000` again.

## Known Constraints

- **Render free tier cold starts:** The backend spins down after 15 minutes of inactivity. First request after idle takes ~30 seconds. Consider upgrading to a paid plan for production.
- **Render free PostgreSQL expires in 90 days.** Upgrade before expiry to avoid data loss.
- **`JWT_SECRET` is generated once.** If Render regenerates it (e.g., you delete and re-create the service), all existing user sessions are invalidated.
- **Single CORS origin.** If you add a custom domain on Netlify, update `FRONTEND_URL` on Render to match the new domain.
